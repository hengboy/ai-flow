# AI Flow - Code Optimize

**触发时机**：用户输入 `/ai-flow-code-optimize`，或明确要求“代码优化”“提升可读性”“安全性加固”“结构整理”“提高可维护性”。

**Announce at start:** "正在使用 ai-flow-code-optimize 技能，在不违背仓库既有架构的前提下执行代码优化。"

## 输入形式

- `/ai-flow-code-optimize <slug> "优化描述"`
- `/ai-flow-code-optimize "优化描述"`

## 规则

> **行为准则**：编码前请遵守 `~/.claude/CLAUDE.md` — 先思考再编码、简洁优先、精准修改、目标驱动执行。

- 若目标 repo 存在 `.ai-flow/rule.yaml`，优化时必须遵守其中的项目级目录边界、禁改文件、必读上下文和验证门禁
- 绑定 `slug` 时，第一执行动作必须是调用 `$HOME/.config/ai-flow/scripts/flow-code-optimize.sh {YYYYMMDD}-{slug}`；该 runtime 入口负责状态允许性判断、review 路由校验和执行前规则门禁

### 1. 优化范围

- 默认优化范围仅包含：可读性、安全性加固、局部结构整理、重复消除、可维护性提升
- 可读性优化包括：命名规范化、函数拆分、消除魔法数字、简化条件表达式
- 结构优化包括：消除重复代码、局部结构整理
- 可维护性优化包括：错误处理规范化、日志规范化
- **不默认包含性能优化**
- 只有用户或 plan 明确要求性能优化时，才允许处理性能相关改动
- 解耦、依赖注入、设计模式引入仅限局部优化场景，且不得改变既有架构、依赖方向、公共接口或跨模块职责
- 如果某项优化需要引入新的核心抽象层、重组模块边界、升级依赖注入骨架，或显著改变调用关系，应转 `/ai-flow-code-refactor`

### 2. 架构硬约束

- 开始前必须先读取 plan 或当前仓库结构，识别既有分层、模块边界、依赖方向、命名风格和抽象层次
- 只能做**架构内优化**
- 不得为了“更优雅”改变模块归属、依赖方向、公共接口、跨仓职责或业务语义
- **避免新增或扩散不合理硬编码**
- 优化过程中不得新增或扩散本应配置化、常量化、枚举化或由上下文推导的常量、路径、环境值、密钥、业务阈值、魔法数字、魔法字符串
- 测试数据、协议明确要求的固定字面量、已有稳定公共常量的直接复用不视为违规
- 如某个值本就应配置化、常量化、枚举化或从现有上下文推导，必须优先复用仓库里已有的配置源、常量定义、领域模型或辅助函数
- 如果当前代码里的问题根因就是硬编码，优化目标应包含消除该硬编码，而不是仅调整写法
- 如果最佳方案需要架构级调整、公共契约变更或需求语义变化，必须立即停止当前优化，回退到 `/ai-flow-plan-coding`；必要时提示用户先走 `/ai-flow-change`

### 3. 绑定模式（有 slug）

- 显式传入 `slug` 时按该状态文件绑定
- 绑定 `slug` 时，如果请求不是单一可直接执行的明确优化动作，也必须先进入“候选项选择模式”，不能直接开始修改
- 候选项生成规则与独立模式一致：同时合并用户诉求拆解项与当前目标范围内识别出的具体优化点，去重后统一编号，并输出优先级、来源、涉及文件或模块
- 候选项列表必须使用 markdown table 展示，避免高占高的长列表；默认列为：`序号`、`优化标题`、`一句话说明`、`优先级`、`来源`、`涉及文件或模块`
- 严禁把候选项输出成“键: 值”逐条列表、分隔线列表或伪表格；特别禁止出现如下形式：
  - `序号: 1`
  - `优化标题: ...`
  - `一句话说明: ...`
  - `────────────────────`
- 若草稿不是 markdown table，必须先重写成 table 再发送给用户
- 绑定模式下同样使用统一优先级：
  - `P1`：安全风险、明显维护性阻塞、错误处理缺陷、会持续放大修改成本的问题
  - `P2`：高频重复、局部结构混乱、命名歧义、可读性显著偏差
  - `P3`：锦上添花型整理、轻量命名微调、低风险格式化式优化
- 未获得用户对候选项的明确选择前，不得执行任何代码修改，也不得推进任何状态

允许进入的绑定状态：

- `AWAITING_REVIEW`
  - 允许执行优化
  - 可先列出候选项，再由用户选择“全部优化”或按序号执行部分优化
  - **不推进状态**
  - 完成后下一步进入 `/ai-flow-plan-coding-review`
- `REVIEW_FAILED`
  - 仅当最近失败报告中的**全部阻塞缺陷**都明确路由到 `ai-flow-code-optimize` 时允许进入
  - 候选项中必须显式区分：
    - 必选项：最近失败报告中的全部阻塞缺陷，以及修复这些阻塞缺陷所必需的最小配套优化
    - 可选项：不影响本轮 review 通过的建议性优化
  - 此场景下候选项 table 必须额外增加 `选择约束` 列，取值仅为 `必选` 或 `可选`
  - 用户可以选择“全部优化”，也可以只选择“必选项 + 部分可选项”；**不得跳过任何必选项**
  - 只有在用户完成选择后，才允许沿用 `/ai-flow-plan-coding` 的修复语义，调用 `flow-state.sh start-fix <slug>` 进入 `FIXING_REVIEW`
  - 完成后调用 `flow-state.sh finish-fix <slug>` 回到 `AWAITING_REVIEW`
- `FIXING_REVIEW`
  - 允许继续执行优化修复
  - 若再次列出候选项，必须继续保留全部未完成的阻塞缺陷为必选项，用户不得跳过
  - 完成后调用 `flow-state.sh finish-fix <slug>` 回到 `AWAITING_REVIEW`
  - 仅当全部必选项已完成时，才允许调用 `flow-state.sh finish-fix <slug>` 回到 `AWAITING_REVIEW`

禁止进入的绑定状态：

- `AWAITING_PLAN_REVIEW`
- `PLAN_REVIEW_FAILED`
- `PLANNED`
- `IMPLEMENTING`
- `DONE`

遇到上述状态时，必须拒绝进入绑定优化，并提示正确回流方向。

### 4. 独立模式（无 slug）

- 不创建、不修改 `.ai-flow/state`
- 若当前修改用于处理 `DONE` 后 review 留下的 Minor 优化建议，应保持无 `slug` 独立执行，不得重新绑定原 plan
- 只允许在用户**明确指定文件、模块、问题范围，或明确授权“扫描整个项目”**时运行
- 默认优先建议用户缩小到文件、目录、模块或关注点；但当用户明确回复“扫描整个项目”“全仓扫描”“扫描全部代码”等同义表达时，必须允许进入**全仓候选项扫描模式**，不得直接拒绝
- 若用户尚未明确扫描范围，第一步必须先进入“范围选择模式”，向用户索要范围；在用户作答前，禁止执行任何 `Explore`、`Read`、`Grep`、`Glob`、运行测试、查看目录树或其他仓库扫描动作
- 范围选择模式下，禁止输出“让我先了解当前仓库结构和待优化候选项”这类会立即开始扫描的表述；只能先请求用户确认范围
- 当用户未绑定 `slug` 直接调用本技能时，且请求不是单一可直接执行的明确优化动作，必须先进入“候选项选择模式”，不能直接开始修改
- 候选项必须同时来自两类来源：
  - 用户输入中可拆解的泛化优化诉求，例如“可读性提升”“结构整理”“安全性加固”
  - 基于当前目标范围实际识别出的具体优化点，例如重复逻辑、魔法数字、错误处理不一致、命名不清晰
- 候选项必须合并去重后统一编号并以 markdown table 输出；每项至少包含：
  - 序号
  - 优化标题
  - 一句话说明
  - 优先级：`P1` / `P2` / `P3`
  - 来源：`用户诉求` / `代码识别` / `两者都有`
  - 涉及文件或模块
- 优先级定义必须显式一致：
  - `P1`：安全风险、明显维护性阻塞、错误处理缺陷、会持续放大修改成本的问题
  - `P2`：高频重复、局部结构混乱、命名歧义、可读性显著偏差
  - `P3`：锦上添花型整理、轻量命名微调、低风险格式化式优化
- 输出候选项后，必须明确交由用户选择：
  - `全部优化`
  - `部分优化：按序号选择，如 1,3,5`
- 未获得用户选择前，不得执行任何代码修改
- 若某候选项超出“架构内优化”边界，必须从候选列表中剔除，并明确提示应转 `/ai-flow-code-refactor` 或 `/ai-flow-plan-coding`
- 独立优化完成后，如需审查，统一提示使用 `/ai-flow-plan-coding-review`；若要保持独立链路，显式使用 standalone review

### 4.1 候选项输出示例

候选项选择模式必须直接参考以下格式输出：

- 输出候选项时，必须先写完整表头行和分隔行，再逐行填充数据
- 即使终端较窄、列内容较长、markdown table 可能换行显示，也**不得**退回成“键: 值”列表
- 若内容过长，可适度缩短“一句话说明”或将文件路径压缩到必要层级，但必须保留 table 结构
- 候选项之间不得插入空行、长横线分隔符或额外的小标题
- 若最终文本不包含表头 `| 序号 | 优化标题 |`，则视为输出不合规，必须重写

```markdown
| 序号 | 优化标题 | 一句话说明 | 优先级 | 来源 | 涉及文件或模块 |
| --- | --- | --- | --- | --- | --- |
| 1 | 异常处理器泄漏异常消息到客户端 | `handleUnexpected` 返回内部异常消息，可能泄露实现细节 | P1 | 代码识别 | `AuthExceptionHandler.java` |
| 2 | 微信 `appSecret` 缺乏序列化保护 | 敏感配置字段可能进入日志或序列化输出 | P1 | 代码识别 | `WechatMiniProperties.java` |
```

若处于 `REVIEW_FAILED` / `FIXING_REVIEW` 且存在必选项，使用以下格式：

```markdown
| 序号 | 优化标题 | 一句话说明 | 优先级 | 来源 | 选择约束 | 涉及文件或模块 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 修复 review 阻塞缺陷 A | 消除阻塞当前 review 通过的问题 | P1 | 代码识别 | 必选 | `src/a.ts` |
| 2 | 顺手整理重复校验逻辑 | 不影响本轮 review 通过，但可减少重复 | P2 | 两者都有 | 可选 | `src/b.ts` |
```

### 4.2 范围选择模式

当用户未给出明确文件、模块或问题边界时，必须先停在这里，不得继续扫描仓库。允许的首条引导应类似：

```text
正在使用 ai-flow-code-optimize 技能，在不违背仓库既有架构的前提下执行代码优化。

请先指定这次要扫描的范围，我会在该范围内列出候选优化项 table。可直接回复其中一种：
1. 单个文件：如 `src/auth/AuthExceptionHandler.java`
2. 单个模块/目录：如 `src/auth`
3. 多个明确路径：如 `src/auth, src/common/security`
4. 已知问题点：如 “只看异常处理和敏感配置序列化”
5. 扫描整个项目：我会先做全仓候选项扫描，并只输出候选项 table 供你选择，不会直接修改代码
```

不合规示例：

```text
正在使用 ai-flow-code-optimize 技能，在不违背仓库既有架构的前提下执行代码优化。

让我先了解当前仓库结构和待优化的候选项。
Explore(Explore isp-auth for optimizations)
```

发送前必须自检：

- 若用户还没有确认范围，而你的下一步包含 `Explore(`、`Read(`、`Grep(`、`Glob(` 或任何仓库扫描动作，说明流程违规，必须先改成范围选择模式
- 只有在用户明确回复了文件、目录、模块或问题边界后，才允许开始扫描并生成候选项 table

### 4.3 全仓候选项扫描模式

当用户明确授权“扫描整个项目”“全仓扫描”“扫描全部代码”时，允许扫描整个仓库，但只限于生成候选项，不得直接开始修改。

进入该模式后的首段说明应类似：

```text
正在使用 ai-flow-code-optimize 技能，在不违背仓库既有架构的前提下执行代码优化。

你已选择扫描整个项目。我会先做全仓候选项扫描，归并重复问题并按优先级输出候选项 table。本阶段只扫描和整理候选项，不会直接修改代码。扫描完成后，你再选择“全部优化”或按序号执行部分优化。
```

全仓模式下必须遵守：

- 只允许输出候选项 table，不得因为范围大就直接进入改代码阶段
- 必须对重复或相近问题做归并，避免输出过长的碎片化候选项
- 候选项过多时，应优先保留高优先级项，并将低优先级相似项合并到一行说明中
- 扫描完成后，必须明确要求用户选择：
  - `全部优化`
  - `部分优化：按序号选择，如 1,3,5`

### 5. 与其他技能的边界

- 需要修复功能错误、逻辑错误、契约错误、需求偏差时，应回到 `/ai-flow-plan-coding`
- 需要架构级重构、模块搬迁、依赖方向调整、公共接口重塑时，应优先考虑 `/ai-flow-code-refactor`
- 解耦如果涉及模块级重组、接口层升级、依赖注入框架化，不属于本技能范围
- 设计模式应用如果会新增系统级抽象、提高整体复杂度或改变主要调用关系，不属于本技能范围
- 只有在**既有架构内**做优化类收敛时，才应使用本技能

## 固定输出协议

完成后先给出自然语言摘要。面向用户的最终回复应在摘要后附一张 markdown table 汇总结果；该 table 用于继续承载优化项展示，避免回退到高占高列表。结果 table 默认列为：`序号`、`优化标题`、`优先级`、`涉及文件或模块`、`完成状态`、`结果说明`。

- `完成状态` 列必须使用显式符号：
  - 成功：`✅`
  - 失败：`❌`
- 若最终结果为 `partial`，结果 table 中必须同时出现 `✅` 与 `❌`，并在摘要中说明哪些项失败、是否需要用户决定后续动作
- 结果汇总同样严禁退回到“`序号: 1` + 多行字段”的键值列表；只允许输出 markdown table
- 即使结果说明较长，也只能缩短单元格内容，不能改成逐项分段说明
- 除结果 table 与必要的下一步提示外，禁止直接向用户输出 `RESULT:`、`ARTIFACT:`、`STATE:`、`NEXT:`、`SUMMARY:`、`SCOPE:`、`INCOMPLETE:` 等协议字段；只有用户明确要求查看协议字段时才可额外展示。内部仍需保留协议块供自动化解析。

本技能协议在仓库统一协议基础上补充 `SCOPE`；`ARTIFACT` 固定为 `none`。

结果汇总格式示例：

```markdown
已完成所选优化项，其中 1 项成功，1 项失败，失败项需要继续确认处理方式。

| 序号 | 优化标题 | 优先级 | 涉及文件或模块 | 完成状态 | 结果说明 |
| --- | --- | --- | --- | --- | --- |
| 1 | 异常处理器泄漏异常消息到客户端 | P1 | `AuthExceptionHandler.java` | ✅ | 已改为返回通用错误信息，并保留服务端日志 |
| 2 | 微信 `appSecret` 缺乏序列化保护 | P1 | `WechatMiniProperties.java` | ❌ | 当前配置模型被外部组件复用，需先确认序列化契约后再改 |
```

发送前必须自检：

- 若输出里出现 `序号:`、`优化标题:`、`一句话说明:` 这类行首键值格式，说明格式不合规，必须改写为 table
- 若候选项或结果汇总没有表头行（例如 `| 序号 | 优化标题 |`），说明格式不合规，必须重写

```text
RESULT: success|failed|partial
AGENT: ai-flow-code-optimize
ARTIFACT: none
SCOPE: bound|standalone
STATE: <status|none>
NEXT: ai-flow-plan-coding-review|ai-flow-plan-coding|ai-flow-change|none
SUMMARY: <one-line-summary>
INCOMPLETE: <仅 RESULT=partial 时输出>
```
