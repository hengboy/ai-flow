## 输入类型处理

- 如果”需求描述”是一份完整实施方案（例如已经包含目标、文件边界、实施步骤、测试计划、风险、验收标准等），必须把它当作上游方案素材来抽取和重写，不得照搬原方案的章节、编号或格式。
- 对完整实施方案输入，只保留其可执行事实：目标、非目标、范围、文件路径、步骤顺序、依赖、风险、验证命令、验收标准；然后全部映射到下方 `plan-template.md` 的 8 章结构。
- 如果原方案已有步骤或验证命令，应优先沿用其中明确、可执行、与当前项目技术栈匹配的内容；缺少的 `2.6 高风险路径与缺陷族`、`4.4 定向验证矩阵`、Step review 关注面、关闭条件、阻塞条件必须按模板补齐。
- 如果”需求描述”只包含一个或多个文档地址（URL、绝对/相对文件路径、知识库链接、Issue/Jira 链接等），`**原始需求（原文）**` 字段可以直接引用这些文档地址原文；不要凭空扩写文档内容。若当前上下文无法读取文档内容，必须在 plan 中把读取/确认文档内容作为前置阅读或阻塞条件，而不是臆测需求细节。
- 如果”需求描述”同时包含文档地址和补充文字，`**原始需求（原文）**` 必须完整保留输入原文；可在目标、背景、步骤中基于补充文字和可见上下文提炼实施计划。

## 模板结构硬门禁

**以下规则优先于所有其他要求，不得违反。**

### 章节门禁
- **必须且只能**输出模板中定义的以下顶级章节，顺序不可改变，编号不可修改：
  `## 1. 需求概述` → `## 2. 技术分析` → `## 3. 实施步骤` → `## 4. 测试计划` → `## 5. 风险与注意事项` → `## 6. 验收标准` → `## 7. 需求变更记录` → `## 8. 计划审核记录`
- **禁止**添加、删除、重命名任何顶级章节（## N）或带编号的子章节标题（### N.N）
- **禁止**创建模板中不存在的章节（如”实施时间线””附录””配置变更汇总””DDL 变更汇总”等），需要这些内容时写入对应已有章节内
- 模板中的 `### 2.1` 到 `### 2.6`、Step 用到的普通 `### 标题`、`### 4.1` 到 `### 4.4`、`### 8.1` 到 `### 8.3` 子章节结构必须完整保留
- 可以在 Step 之间增加 Phase/Stage 分组标题（如 `---` 分隔线 + 简短说明），但 **不得改变 Step 自身的标题结构和内部子字段结构**

### 首行门禁
- 首行必须是 `# 实施计划：{一句话标题}`，不得省略”实施计划：”前缀，不得使用表格或其他格式替代

### 元数据门禁
- 文件头部元数据必须使用 `> ` 引用块格式（与模板一致），不得使用表格、列表或其他格式

### Step 子字段门禁
每个 Step（`### 标题`）内部必须按顺序包含以下子字段，**不可改名、不可改变层级**：
- `**Step ID**：`（必须，使用稳定 slug 风格标识，全文唯一）
- `**目标**：`（必须）
- `**文件边界**：`（必须，至少一行 `- Create:` / `- Modify:` / `- Test:`）
- `**本轮 review 预期关注面**：`（必须）
- `**前置阅读**：`（不需要时整段删除，不可保留空占位）
- `**执行动作**：`（必须包含 `- [ ]` 复选框）
  - **默认包含** `写失败用例`、`运行失败验证`、`实现最小改动`、`运行通过验证`、`本步自检` 五个子项
  - **简化允许**：对于明确的单行修改 / 纯配置变更（如改默认值、改一行类型），可合并为 `实现` + `验证` + `自检` 三项，省略失败用例环节
  - 每个动作必须给出确切命令和预期结果
- `**本步验收**：`（必须，`- [ ]` 复选框，至少一条）
- `**本步关闭条件**：`（必须）
- `**阻塞条件**：`（没有明确阻塞条件时写"- 无"，不可删除该字段）

### 裁剪规则
- `### 2.2 数据模型变更`：不涉及数据库变更时写"不涉及数据库变更"，不可删除子章节
- `### 2.3 API 变更`：不涉及 API 变更时保留表头，表体写"无新增/修改接口"，不可删除子章节
- Step 内的 `**前置阅读**`：不需要时整段删除
- 其他模板中定义的子章节和字段必须保留

### 内容填充要求
1. 先做头脑风暴式 intake：主动识别需求中的不确定项、潜在分歧点和会影响决策的边界条件；确认目标、非目标、约束、成功标准、仓库边界与验证方式
2. 只要存在任何不确定项，必须先询问用户，不得自行猜测后直接写入 plan；询问时必须同时列出可行性、推荐选项和备选项，供用户交互式选择
3. 如果需求包含多个独立子系统，必须先拆解边界和推荐顺序，再让用户确认优先级与范围
4. 填充模板中的所有占位符，删除大括号标记
4.1. `## 1. 需求概述` 中必须保留 `**原始需求（原文）**` 字段，并完整引用需求描述原文，不得改写；仅当输入是一个或多个文档地址时，可以直接引用这些文档地址作为原文
4.2. `## 1. 需求概述` 中的 `**目标**`、`**背景**`、`**非目标**` 字段必须填充，不可省略
5. 实施步骤要拆得足够细，每个 Step 先写 `Step ID`、目标和文件边界（file-boundary），文件路径必须精确，并补齐”本轮 review 预期关注面””本步关闭条件”
6. 代码必须严格匹配上述检测到的技术栈，不要引入项目不使用的语言或框架
7. 直接输出完整的计划 Markdown 内容，不要包含其他解释文字
8. 本计划是证据文档，不承担流程状态语义；流程状态只写入 .ai-flow/state/{YYYYMMDD}-__AI_FLOW_SLUG__.json
   - 不得为 .ai-flow/state/{YYYYMMDD}-__AI_FLOW_SLUG__.json 设计任何 JSON 结构、字段列表、样例内容或手工维护步骤
   - 该状态文件只能由 flow-state.sh 维护，固定字段只有：schema_version、slug、title、current_status、created_at、updated_at、plan_file、review_rounds、latest_regular_review_file、latest_recheck_review_file、last_review、active_fix、transitions
   - 如果需要记录步骤进度、验证结果、变更登记，写在计划文档自身（Step 复选框、测试计划、需求变更记录），不要写进 state JSON 设计
9. 所有规则、算法、接口结构必须自包含，不能引用 plan 外的代码或文档
10. “**前置阅读**”字段是可选的，仅在需要参考现有代码范式（如新增文件要遵循已有模式、修改接口需了解调用方）时填写，说明为什么要读和了解什么
11. 每个 Step 的”执行动作”必须使用 “- [ ]” 复选框，动作粒度控制在 2-5 分钟（step-run）
12. `2.5 文件边界总览` 的“对应 Step ID”列必须引用对应 Step 的 `Step ID`，不得写 `Step 1` 之类的可见序号
13. 涉及代码修改的动作必须说明具体函数、类型、配置、分支逻辑、错误处理和边界条件
14. 技术分析必须包含”2.6 高风险路径与缺陷族”：每个高风险能力至少写清影响面、典型失效模式、对应缺陷族和必须覆盖的验证方式
15. 测试计划必须包含”4.4 定向验证矩阵”：每个缺陷族至少绑定一个定向验证命令；SQL-heavy / mapper-heavy / workflow-heavy 需求至少覆盖 test-compile、目标单测、目标 Mapper/集成验证中的两类
16. 缺陷族至少要能追踪到数据/SQL/映射、权限/范围、状态机/流程、输入/边界、测试/证据这些常见风险中的相关项；如果某类不适用，要明确写”不适用”原因
17. 验证动作必须给出确切命令和预期结果；测试优先（test-first）按”写失败用例 -> 验证失败 -> 最小实现 -> 验证通过”组织
18. 每个 Step 必须包含”本步自检”，至少检查 git diff 范围、计划外文件、占位符和临时日志
19. 禁止输出 TBD、TODO、后续补充、类似上一步、适当处理异常、根据情况处理等不可执行描述
20. `## 8. 计划审核记录` 必须保留完整结构，且初始内容保持”待审核 / 不允许进入 execute”的 draft 语义
21. 如果某类内容不适用，删除对应模板行或写”无”，不要保留占位符
22. 输出前自检：需求覆盖完整、无占位符、函数/类型/路径命名一致、2.6 和 4.4 已填充、每个 Step 都有验证命令和预期结果、`原始需求（原文）` 与 `## 8` 已保留
23. Plan 参与仓库规则：

   - 如需求明确涉及 owner 旁边的本地 Git 仓库，给该仓库分配稳定 repo id，并在 `2.1 涉及模块` 与 `2.5 文件边界总览` 的 `仓库` 列填写该 id。
   - 跨仓文件路径必须写成 `repo_id/path/to/file`；owner 仓库可写 `owner/path/to/file` 或项目内相对路径。
   - Git 验证命令必须对相应仓库使用 `git -C <repo_path>` 前缀，其中 owner 为 `.`，参与仓库为相对 owner repo 的路径。
